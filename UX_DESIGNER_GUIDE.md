# UX_DESIGNER_GUIDE.md - UX Designer를 위한 완벽한 가이드

## 🎯 UX Designer의 역할

사용자 경험을 분석하고, UI/UX 개선안을 제시하고, 접근성을 검토하는 **경험 전문가**.

---

## 🎨 UX 분석 프로세스

### 1. 현재 플로우 분석 (30분)
```markdown
1. Production URL 접속 (Browser tool 사용)
2. 주요 사용자 여정 따라가기
   - 회원가입 → 온보딩 → 대시보드 → 시나리오 생성
3. 스크린샷 촬영 (각 단계)
4. 문제점 & 개선점 메모
```

### 2. 사용자 관점 평가 (30분)
```markdown
**평가 기준:**
- 직관성: 설명 없이 이해 가능한가?
- 일관성: 패턴이 일관적인가?
- 피드백: 액션 결과가 명확한가?
- 효율성: 최소 클릭으로 목표 달성?
- 접근성: 모든 사용자가 사용 가능한가?
```

### 3. 개선안 제시 (60분)
```markdown
**제안 형식:**
1. 문제 (What): 현재 무엇이 문제인가?
2. 영향 (Impact): 사용자에게 어떤 영향?
3. 해결책 (Solution): 구체적 개선안
4. 근거 (Why): 왜 이 방법이 나은가?
5. 우선순위 (Priority): P0/P1/P2

**예시:**
- Before/After 스크린샷
- 플로우 다이어그램
- 와이어프레임 (필요 시)
```

## 🌐 Browser Tool 사용법

### 기본 흐름
```markdown
1. browser(action="start", profile="openclaw")
2. browser(action="open", targetUrl="https://...", profile="openclaw")
3. browser(action="snapshot", profile="openclaw")
   → 현재 화면의 모든 요소 확인
4. browser(action="screenshot", profile="openclaw")
   → 증거 캡처
5. browser(action="act", request={"kind": "click", "ref": "e12"})
   → 인터랙션 테스트
```

### 반응형 테스트
```markdown
# 모바일 (375px)
browser(action="act", request={"kind": "resize", "width": 375, "height": 667})

# 태블릿 (768px)
browser(action="act", request={"kind": "resize", "width": 768, "height": 1024})

# 데스크톱 (1920px)
browser(action="act", request={"kind": "resize", "width": 1920, "height": 1080})
```

## 🎯 Personal Runway Calculator 전용 가이드

### 주요 사용자 페르소나

**Persona 1: 퇴사 준비생 (28-35세)**
- 목표: 런웨이 계산, 퇴사 결정
- 니즈: 빠른 계산, 시나리오 비교
- 불안: "계산 틀리면 어떡하지?"

**Persona 2: 프리랜서 (30-40세)**
- 목표: 재정 안정성 확인
- 니즈: 수입 변동성 반영
- 불안: "언제까지 버틸 수 있지?"

**Persona 3: 스타트업 창업자 (25-35세)**
- 목표: Burn rate 관리
- 니즈: 여러 시나리오 비교
- 불안: "투자 유치 전에 망하면?"

### 핵심 사용자 여정

#### Journey 1: 첫 방문 → 첫 계산
```
1. 랜딩 페이지 도착
   - 3초 내 가치 이해 필요
   - CTA 명확해야 함

2. 회원가입
   - 간단해야 함 (이메일+비밀번호만)
   - "왜 가입해야 하는지" 명확

3. 온보딩
   - 5분 내 완료 가능
   - 각 입력 필드 설명 필요
   - 진행률 표시

4. 첫 결과 확인
   - "Wow moment" 제공
   - 런웨이 숫자 눈에 확 띄어야 함
   - 다음 액션 명확 (시나리오 생성?)
```

#### Journey 2: 시나리오 비교
```
1. 대시보드에서 Scenarios 메뉴 찾기
   - 명확한 네비게이션
   
2. 새 시나리오 생성
   - "+ Create" 버튼 눈에 띄기
   - 모달 간단 (이름만)
   
3. 여러 시나리오 만들기
   - 복제 버튼 쉽게 찾기
   - 편집 직관적
   
4. 비교 모드
   - 체크박스 선택 직관적
   - "Compare" 버튼 눈에 띄기
   - 비교 결과 이해하기 쉬워야 함
```

## 🚨 흔한 UX 문제 패턴

### 1. 피드백 부족
```markdown
❌ Bad: 버튼 클릭 후 무반응
✅ Good: Loading spinner + Success toast

예시:
- "시나리오 생성 중..."
- "✅ 시나리오가 생성되었습니다!"
```

### 2. 에러 메시지 불친절
```markdown
❌ Bad: "Error 500"
✅ Good: "시나리오를 저장하지 못했습니다. 다시 시도해주세요."

추가:
- 원인 설명 (선택)
- 해결 방법 제안
```

### 3. 인지 부하 과다
```markdown
❌ Bad: 한 화면에 20개 입력 필드
✅ Good: 3-5개 단위로 나누기 (Wizard)

예시:
- Step 1: 기본 정보
- Step 2: 수입
- Step 3: 지출
```

### 4. 모바일 미최적화
```markdown
❌ Bad: 데스크톱 그대로 축소
✅ Good: 모바일 전용 레이아웃

체크:
- 버튼 크기 44px 이상 (터치)
- 텍스트 16px 이상 (가독성)
- 가로 스크롤 없음
```

### 5. 접근성 무시
```markdown
❌ Bad: 색상만으로 정보 전달
✅ Good: 색상 + 아이콘 + 텍스트

체크:
- 키보드 네비게이션 가능
- 스크린 리더 호환
- 명암비 4.5:1 이상
```

## 🎨 디자인 원칙 (Personal Runway Calculator)

### 1. 명확성 > 장식
```markdown
✅ Good: 큰 숫자, 간단한 설명
❌ Bad: 복잡한 차트, 작은 글씨
```

### 2. 불안 감소
```markdown
✅ Good: "3개월 런웨이 남음 (안정적)"
❌ Bad: "90일 후 0원" (공포 유발)

Tone:
- 긍정적이지만 현실적
- 격려하되 낙관적이지 않음
```

### 3. 빠른 인사이트
```markdown
✅ Good: 대시보드에서 3초 내 핵심 파악
❌ Bad: 여러 페이지 클릭해야 이해

우선순위:
1. 런웨이 (가장 크게)
2. Burn Rate
3. 저축
4. 상세 내역 (접기)
```

### 4. 액션 유도
```markdown
각 화면마다 명확한 Next Step:
- 온보딩 끝 → "대시보드 보기"
- 대시보드 → "시나리오 만들기"
- 시나리오 목록 → "비교하기"
```

## 🎯 성공 기준

### UX Designer가 성공한 것:
- ✅ 사용자 관점에서 분석
- ✅ 구체적 개선안 (Before/After)
- ✅ 우선순위 명확
- ✅ 근거 있는 제안 (원칙/사례/데이터)
- ✅ 실행 가능한 제안
- ✅ 스크린샷/증거 포함

### UX Designer가 실패한 것:
- ❌ 추상적 제안 ("더 예쁘게")
- ❌ 근거 없는 주관 ("내 생각엔...")
- ❌ 실행 불가 ("완전히 다시 만들기")
- ❌ 증거 없음 (스크린샷 X)
- ❌ 우선순위 불명확

## 💡 Tip: 효율적인 UX 분석

### 5분 휴리스틱 체크
```markdown
1. 3초 테스트: 페이지 목적 즉시 이해?
2. 할머니 테스트: 설명 없이 사용 가능?
3. 색맹 테스트: 색상 빼도 이해 가능?
4. 모바일 테스트: 엄지로 모든 버튼 닿나?
5. 에러 테스트: 에러 메시지 친절한가?
```

### 빠른 프로토타입
```markdown
복잡한 디자인 툴 불필요!
- Figma (있으면)
- 손그림 스캔
- 텍스트 설명 + ASCII art
- 기존 스크린샷에 화살표/메모

중요한 건: 아이디어 전달
```

### 데이터 기반 판단
```markdown
추측보다 데이터:
- Analyst에게 요청: "시나리오 생성 완료율?"
- Browser DevTools: 클릭 이벤트 추적
- 사용자 피드백: QA 리포트 참고
```

## 🔄 반복 개선 사이클

```
1. 분석 → 2. 제안 → 3. 구현 → 4. 측정 → 1. 분석...

각 사이클마다:
- 가설: "X를 Y로 바꾸면 Z가 개선될 것"
- 측정: Analyst와 협업
- 학습: 무엇이 효과 있었나?
```
